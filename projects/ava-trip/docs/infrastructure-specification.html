Vercel, API/Worker VPS, Managed PostGIS, TourAPI 배치 캐시 기반 인프라 구성을 정리한 문서입니다.

source: 01_Planning/Infrastructure_Specification.md

🌐 아바트립 (AVA-Trip) 분리형 인프라 배포 명세서 (Infrastructure Setup Specification)

본 문서는 2026-05-29 기준으로 재정리한 아바트립(AVA-Trip)의 권장 인프라 구성입니다. 기존의 AWS Lightsail 단일 VPS + Docker Compose 전체 탑재 방식은 PostgreSQL/PostGIS, Redis, FastAPI, Celery, LLM 워커를 2GB RAM 안에 모두 넣어야 하므로 안정성이 낮습니다. 새 구성은 Vercel 프론트엔드, API/Worker 실행 노드, Managed PostgreSQL + PostGIS, TourAPI 배치 캐시 파이프라인으로 책임을 분리합니다.

🏗️ 1. 권장 배포 아키텍처 개요

  • Frontend: Vercel에 배포한 Next.js PWA. 지도, 일정 편집, 그룹 초대, SSE 수신, 공유 화면을 담당합니다.
  • API/Worker Node: AWS Lightsail VPS 1대. FastAPI, Celery/RQ Worker, Redis, Caddy만 Docker Compose로 실행합니다.
  • Database: Supabase PostgreSQL + PostGIS 또는 Lightsail Managed PostgreSQL. 장소 검색, 일정, 페르소나, 협상 세션, TourAPI 캐시를 저장합니다.
  • Queue/Cache: MVP 단계에서는 VPS 내부 Redis. 작업 큐, SSE 이벤트 브로커, 짧은 TTL 캐시를 담당합니다.
  • Batch Pipeline: GitHub Actions, 서버 cron, 또는 Worker beat로 TourAPI 데이터를 사전 수집하고 정규화합니다.
  • External Services: TourAPI 4.0, Kakao Map SDK, Gemini API. 앱 요청 시 TourAPI를 매번 직접 호출하지 않고 내부 DB를 우선 조회합니다.

🧭 2. 왜 단일 VPS에서 분리형으로 바꾸는가

  • PostGIS는 핵심 데이터 계층: 장소 반경 검색, 이동 거리 검증, 일정 충돌 검사에 쓰이므로 API/Worker 부하와 분리해야 합니다.
  • LLM 워커는 메모리 변동이 큼: LangGraph/Celery 작업은 순간 메모리 사용량이 커질 수 있어 같은 VPS의 DB 안정성을 위협합니다.
  • TourAPI는 실시간 의존 대상이 아님: 응답 지연, 누락, 호출 제한 가능성이 있으므로 사전 수집 및 로컬 캐시가 기본입니다.
  • Vercel은 프론트에 집중: Next.js PWA, CDN, HTTPS, 정적/동적 UI에 적합하고, 긴 LLM 협상 작업은 별도 Worker에서 처리합니다.

📡 4. 데이터 파이프라인 전략

TourAPI 장소 데이터는 앱 요청 시 실시간 호출하지 않습니다. 제주 장소 미리보기에서 확인한 것처럼 목록, 상세, 이미지, 혼잡도, 연관 관광지, 방문자 통계는 서로 다른 API를 조합해야 하므로 사용자 요청 경로에서 직접 처리하면 응답 시간이 불안정해집니다.

  1. 수집: 지역/카테고리별 장소 목록을 배치로 수집합니다. 예: 제주 관광지, 문화시설, 액티비티, 맛집, 카페, 쇼핑.
  2. 상세 보강: detailCommon, detailIntro, 이미지, 홈페이지, 전화번호, 운영시간을 별도 작업으로 채웁니다.
  3. 정규화: 카테고리, 주소, 좌표, 이미지 URL, 운영시간, 요금, 태그를 내부 스키마로 변환합니다.
  4. 공간 인덱싱: PostGIS geography(Point, 4326) 또는 geometry(Point, 4326) 컬럼과 GiST 인덱스를 둡니다.
  5. 서빙: 앱은 내부 DB를 우선 조회하고, 누락 데이터만 백그라운드 보강 작업으로 요청합니다.
flowchart LR
    TourAPI[TourAPI 4.0] -- batch fetch --> Ingest[Collector Job]
    Ingest -- normalize --> DB[(PostgreSQL + PostGIS)]
    DB -- query --> API[FastAPI]
    API -- HTTPS/SSE --> PWA[Next.js PWA]
    API -- enqueue --> Redis[(Redis)]
    Redis -- task --> Worker[Worker]
    Worker -- enrich / validate --> DB

🐳 5. Docker Compose 범위

VPS의 Docker Compose는 API 실행 계층만 담당합니다. PostgreSQL/PostGIS는 Docker Compose에서 제거하고 managed DB 연결 문자열을 환경변수로 주입합니다.

version: '3.8'

services:
  redis:
    image: redis:7-alpine
    container_name: avatrip-redis
    restart: always
    command: redis-server --appendonly yes
    volumes:
      - redisdata:/data

  web:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: avatrip-web
    restart: always
    depends_on:
      redis:
        condition: service_started
    env_file:
      - .env
    environment:
      - REDIS_URL=redis://redis:6379/0
    ports:
      - "8000:8000"

  worker:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: avatrip-worker
    restart: always
    command: celery -A main.celery_app worker --loglevel=info -c 1
    depends_on:
      - redis
    env_file:
      - .env
    environment:
      - REDIS_URL=redis://redis:6379/0
    # MVP 단계에서는 LLM 비용과 메모리 폭증 방지를 위해 concurrency 1을 기본값으로 둡니다.

  caddy:
    image: caddy:2-alpine
    container_name: avatrip-proxy
    restart: always
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
      - caddy_config:/config
    depends_on:
      - web

volumes:
  redisdata:
  caddy_data:
  caddy_config:

프론트엔드에 노출해도 되는 값과 서버 전용 비밀값을 분리합니다. TourAPI 서비스 키, Gemini API 키, DB URL은 반드시 서버 환경변수로만 보관합니다.

🔒 7. 리버스 프록시 설정 (Caddyfile)

Caddy는 API 도메인의 HTTPS 종료와 FastAPI 컨테이너 프록시만 담당합니다. DB와 Redis 포트는 외부에 열지 않습니다.

# 아래 도메인은 구매한 도메인이나 서브도메인으로 변경하여 매칭합니다.
api.avatrip.co.kr {
    reverse_proxy web:8000

    header {
        X-Frame-Options "DENY"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
    }
}

🚀 8. 배포 및 운영 워크플로우

8-1. 초기 셋업

  1. Managed PostgreSQL에 PostGIS extension을 활성화합니다.
  2. 장소/이미지/혼잡도/방문자 통계/협상 세션 테이블을 마이그레이션합니다.
  3. TourAPI 제주 seed batch를 실행해 핵심 장소 데이터를 채웁니다.
  4. Lightsail VPS에 API/Worker/Redis/Caddy를 배포합니다.
  5. Vercel 환경변수에 API URL을 연결하고 PWA를 배포합니다.

8-2. 서비스 가동

docker compose pull
docker compose up -d --build

docker compose ps

8-3. 모니터링

docker compose logs -f web

docker compose logs -f worker

docker compose logs -f redis

8-4. 운영 체크리스트

  • DB 백업: Managed DB 자동 백업 또는 일별 dump를 설정합니다.
  • API 상태 확인: /health, /ready 엔드포인트를 분리합니다.
  • 작업 실패 재시도: TourAPI 수집 및 LLM 협상 작업에 retry/backoff를 둡니다.
  • Rate Limit: 초대 링크, 협상 시작, 장소 상세 보강 API에 사용자/IP 기준 제한을 둡니다.
  • 로그 분리: API access log, worker job log, batch ingest log를 분리해 장애 원인을 추적합니다.
