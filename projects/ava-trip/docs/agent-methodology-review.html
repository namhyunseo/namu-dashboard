Hermes 방식과 Harness 방식의 조율 구조를 비교한 기술 검토 보고서입니다.

📑 Hermes Agent & Harness Engineering 기술 분석 및 검토서

본 문서는 아바트립(AVA-Trip) 멀티 에이전트 시스템 설계에 있어 핵심 후보로 검토 중인 Hermes Agent(Nous Research)와 Harness Engineering(Mitchell Hashimoto) 두 기술에 대한 정확한 정의를 정립하고, 각각의 개별적 타당성 검토 및 향후 융합 시너지 모델을 제안합니다.

🔍 1. 기술 재정의 및 기존 오해 정정

이전 설계 초안에서 발생한 개념적 혼선 및 용어의 오해를 다음과 같이 바로잡습니다.

① Hermes Agent (Nous Research)

  • 오해: 단순한 비동기 이벤트 분산 메시징 처리 파이프라인으로 간주되었습니다.
  • 사실: Nous Research가 2026년 2월 공식 출시한 오픈소스 자율 AI 에이전트 시스템(https://hermes-agent.org/ko/)입니다.
  • 사용자의 선호도와 과거 이력을 모든 세션에서 유지하는 지속 메모리(Persistent Memory)를 지원합니다.
  • 에이전트가 문제를 해결하는 과정에서 재사용 가능한 도구를 스스로 개발하여 문서화하는 자동 스킬 생성(Automated Skill Creation, SKILL.md 포맷) 능력을 갖추고 있습니다.
  • Telegram, Discord, Slack 등 여러 플랫폼과 연결할 수 있는 멀티플랫폼 게이트웨이를 기본 탑재하고 있습니다.

② Harness Engineering (하네스 엔지니어링 - Mitchell Hashimoto)

  • 오해: 게임이론 기반의 수학적 내시 균형 솔버인 "HANES"로 해석되었습니다.
  • 사실: HashiCorp의 창업자 Mitchell Hashimoto가 제안한 에이전트 시스템 구축 패러다임(Agent = Model + Harness)입니다.
  • 확률적으로 동작하여 환각(Hallucination)이 잦은 LLM(Model) 단독으로는 완벽한 비즈니스 로직을 수행할 수 없다는 문제에서 출발합니다.
  • LLM 외부를 둘러싸는 단단한 소프트웨어 장치인 하네스(Harness)를 구축하여, 입력/출력 스키마를 검증하고, 도메인 제약조건을 물리적으로 검사(Linter)하며, 오류 발생 시 모델이 스스로 수정한 후 올바른 값을 출력할 때까지 피드백 루프(Self-Correction)를 돌리는 구조입니다.

🧠 2. 개별 기술의 아바트립(AVA-Trip) 적용 타당성 검토

두 개의 에이전트 기술이 아바트립 서비스의 개별 요구사항을 해결하는 데 있어 얼마나 적절한지 분석합니다.

2-1. Hermes Agent (Nous Research) 검토

아바트립은 "그룹 여행자들의 성향(페르소나)을 대리하여 에이전트들이 상호 협상하는 시스템"입니다. 이 관점에서 Hermes Agent의 특성을 평가합니다.

👍 긍정적 측면 (도입 시의 이점)

  1. 지속적 페르소나 메모리: 사용자가 과거에 작성한 일정 피드백, 좋아하는 여행 스타일(예: "활동적인 액티비티 선호", "인스타 감성 카페 필수")을 에이전트가 완벽히 기억하고 시간이 지날수록 고도화된 개인화 서비스를 제공할 수 있습니다.
  2. 스킬의 자동 확장: 여행지 TourAPI 조회, 날씨 API 조회, 카카오맵 경로 검증 등 여행 계획에 필요한 새로운 기능(Tool)을 에이전트가 스스로 SKILL.md 형태의 규격화된 코드로 구성하고 실행할 수 있습니다.
  3. 멀티플랫폼 확장성: 사용자가 PWA 앱뿐만 아니라, Telegram이나 Discord 단체방에 아바트립 에이전트를 초대하여 실시간으로 일정을 추천받고 조율하는 강력한 사용자 경험(UX)으로 확장이 매우 용이합니다.

⚠️ 제약 및 고려사항 (보완이 필요한 점)

  • 리소스 오버헤드: Hermes Agent는 독자적인 CLI 백엔드와 데몬 프로세스 구조를 가지고 있어, 모바일 PWA 환경에서 다수의 일반 사용자를 대상으로 매 세션마다 온전한 Hermes 인스턴스를 무겁게 띄우는 것은 비효율적일 수 있습니다.
  • 적용 방향: Hermes Agent의 풀 에이전트 데몬을 그대로 사용하는 대신, 해당 아키텍처가 가진 (1) 지속적 메모리 저장 체계 및 (2) SKILL.md 규격의 도구 컴파일 표준을 아바트립 AI 백엔드(FastAPI + Celery) 내에 경량 모델로 이식하거나 헤드리스(Headless) 워커 형태로 활용하는 것이 적절합니다.

2-2. Harness Engineering (하네스 엔지니어링) 검토

여행 계획 서비스는 "물리적/정량적 제약 조건이 극도로 깐깐한 도메인"입니다. (예: 이동거리의 한계, 운영 시간 매칭, 예산 한도 준수, 일정의 시간적 순서 보장 등).

👍 긍정적 측면 (도입 시의 이점)

  1. 환각(Hallucination)의 원천 방지: LLM은 두 관광지의 실제 거리나 차량 이동 시간을 임의로 지어내기 쉽습니다. 하네스가 이 출력을 가로채어 PostGIS(지리 데이터 연산) 및 카카오맵 API를 통해 이동 시간이 3시간을 초과하거나 불가능한 동선인 경우 즉각 감지합니다.
  2. 자가 교정(Self-Correction) 피드백 루프: 하네스가 오류(예: *"A 카페는 화요일 휴무인데 화요일 일정에 배치됨"*)를 발견하면, 이를 정중하고 구체적인 피드백 메시지로 구성하여 에이전트에 다시 입력합니다. 모델은 이 피드백을 기반으로 실수를 인정하고 올바른 대안 일정을 재생성합니다.
  3. 예산/비용 Linter: 에이전트가 합의한 총비용이 사용자가 설정한 예산 한도(ex. 1인당 20만 원)를 초과할 경우 하네스가 이를 거절하고 다른 저렴한 스팟으로 변경하도록 강제합니다.

⚠️ 제약 및 고려사항

  • 지연 시간(Latency) 증가: 자가 교정 피드백 루프가 여러 번 돌게 되면 LLM API 호출 횟수가 늘어나 사용자의 대기 시간이 길어질 수 있습니다.
  • 적용 방향: 피드백 루프의 최대 시도 횟수를 엄격히 제한(최대 2~3회)하고, 최종 실패 시에는 하네스가 수학적 보정 알고리즘(예: 예산 초과 시 마지막 우선순위 관광지를 자동으로 일정에서 탈락시키는 등)을 적용해 확정적인 출력을 최종 보장해야 합니다.

🤝 3. Hermes × Harness 융합 시너지 모델 (Conceptual Architecture)

두 기술을 결합하여 "지속적이고 똑똑한 개성을 가진 에이전트(Hermes)가 소프트웨어 하네스(Harness)의 안전망 안에서 확실하게 동작하는 구조"를 실현할 수 있습니다.

flowchart TD
    subgraph UserGroup [사용자 협상 그룹]
        UserA[사용자 A - PWA]
        UserB[사용자 B - PWA]
    end

    subgraph HermesCore [Hermes Agent 레이어 - 에이전트 개성 및 도구]
        AgentA[Hermes 에이전트 A<br/>- 사용자 A 메모리 탑재<br/>- 선호도/과거 이력 학습]
        AgentB[Hermes 에이전트 B<br/>- 사용자 B 메모리 탑재<br/>- 선호도/과거 이력 학습]
        
        AgentA <-->|비동기 조율 및 타협 제안| AgentB
    end

    subgraph HarnessSystem [Harness Engineering 레이어 - 안전망 및 검증]
        SchemaVerifier[1. Schema Validator<br/>Pydantic JSON 포맷 보장]
        TravelLinter[2. Travel Linter<br/>- PostGIS 공간 동선 검증<br/>- 운영시간/휴무일 대조<br/>- 총 예산 초과 여부 확인]
        SelfCorrection[3. Self-Correction Loop<br/>오류 발견 시 에이전트에 피드백 전달]
    end

    subgraph CoreEngine [데이터 및 도구 엔진]
        PostGIS[(PostgreSQL + PostGIS)]
        TourAPI[TourAPI 4.0 / Kakao Map]
    end

    %% 데이터 흐름
    UserA -->|여행 성향 입력| AgentA
    UserB -->|여행 성향 입력| AgentB
    
    AgentA & AgentB -->|초안 일정 합의문 작성| SchemaVerifier
    SchemaVerifier -->|올바른 포맷 완료| TravelLinter
    
    TravelLinter <-->|공간 및 공공데이터 쿼리| CoreEngine
    
    TravelLinter -->|동선 오류 / 예산 초과 감지| SelfCorrection
    SelfCorrection -.->|오류 내용 피드백 및 재작성 지시| HermesCore
    
    TravelLinter -->|100% 무결점 보장| FinalOutput[최종 무결점 일정표 확정]
    FinalOutput -->|사용자 브라우저 스트리밍 렌더링| UserGroup

융합 파이프라인의 이점

  1. 안전성 + 개인화의 공존: 사용자의 주관적인 성향(Hermes 메모리)을 충족하면서도, 실제 여행 시 발생할 수 있는 시간적/공간적 오류(Harness 검증)가 0%인 완벽한 여행 계획이 수립됩니다.
  2. 컴파일된 여행 스킬(Tool)의 신뢰성: 에이전트가 자동 생성한 SKILL.md (예: "특정 카테고리 음식점 탐색 툴")이 실행될 때, 하네스 내부의 샌드박스에서 검증되므로 시스템 전반의 보안과 안정성이 보장됩니다.
