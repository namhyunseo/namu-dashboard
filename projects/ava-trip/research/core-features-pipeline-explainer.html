AVA-Trip Core Engine

3대 핵심 기능 및 백엔드 파이프라인 명세

페르소나 생성, 에이전트 협상(A/B 모델 비교), 하네스 자가 교정 및 전반적인 시스템 아키텍처 가이드

1 페르소나 기반 에이전트 생성

"에이전트 생성"은 단순히 고정된 정적 텍스트를 불러오는 것이 아닙니다. 사용자가 가입 시 진행한 여행 성향 설문조사, 과거 여행 이력, 특정 키워드(예산, 이동거리, 식성) 만족도를 데이터베이스(PostgreSQL)에서 실시간으로 조회하여 LLM 시스템 프롬프트(System Prompt)로 컴파일하고 바인딩하는 파이프라인입니다.

🗄️ PostgreSQL 사용자 프로필
{ "user_id": "usr_9921", "name": "서연", "preferences": { "theme": "healing_cafe", "activity_level": "low", "budget_limit_daily": 100000 }, "constraints": { "avoid_crowds": true, "max_travel_km": 40 } }
➡️ Compile
🤖 에이전트 주입용 System Prompt
# Persona Rule 당신의 이름은 [서연]입니다. 당신은 조용하고 고즈넉한 숲속 카페에서 피톤치드를 마시며 힐링하는 것을 아주 선호합니다. # Constraints - 하루 총 예산: 절대 100,000원을 초과해서는 안 됩니다. - 이동 거리: 1일 최대 이동 반경은 40km 이내여야 합니다. # Mission 당신은 민수, 준호와 협상하여 최적의 절충 일정을 도출해야 합니다. 만약 이 조건을 위반하는 초안이 만들어지면 절대 타협하지 마십시오.
엔지니어링 핵심: 아바트립 백엔드는 다수의 에이전트를 가볍게 돌리기 위해 매 사용자 접속마다 가상 운영체제나 무거운 CLI 데몬을 띄우지 않습니다. 대신, Redis 캐시 세션에 위 콘텍스트 카드를 임시 적재하고 LLM API 호출 시 필요할 때마다 동적으로 실시간 주입하는 헤드리스(Headless) 에이전트 방식을 사용합니다.

2 에이전트 간 대리 협상 (Option A & B 비교)

각자의 선호도가 명확한 3명의 취향 카드가 생성되었을 때, 이들의 요구조건을 조율하여 1차 초안 일정을 설계하는 협상 메커니즘입니다. 시스템 설계 관점에서 Option A(다중 턴 롤플레이)와 Option B(단일 오케스트레이터 일괄 시뮬레이션) 두 가지 접근법이 존재합니다.

Option A: Multi-Agent Turn Debates

각 사용자 아바타를 개별 프롬프트를 가진 LLM 인스턴스로 각각 호출하여, 실제 메신저 단톡방처럼 번갈아 대화하게 하여 절충안을 도출하는 턴제 토론 방식입니다.

사용자 경험 (협상 과정의 극적 묘사) 매우 우수 (95점)

API 토큰 비용 매우 높음 (90점)

API 지연 시간 (Latency) 매우 느림 (85점 / 약 10~15초)
🔄 에이전트 A ➡️ 에이전트 B ➡️ 에이전트 C 순서로 연속 API 호출 [API call 1] A: "삼척 해변 서핑이 어때요?" [API call 2] B: "거리는 좋은데 예산 10만원 이하로 해야 합니다." [API call 3] C: "대게를 먹으면 비용이 넘어가니 물회로 바꿉시다." [API call 4] A: "합의안 도출 완료! 확정합시다." ※ 총 4~5회 이상의 연속 LLM 체인 호출 필요
Option B: Single Orchestrator Joint Simulation

하나의 강력한 LLM 프롬프트 내에 3명의 페르소나 카드와 상황을 동시에 주입하고, 한 번의 API 호출로 "이 세 명의 가상 대리 대화 내용"과 "최종 도출된 일정 스키마"를 원샷(One-Shot)으로 즉시 출력받는 방식입니다.

사용자 경험 (과정 묘사 속도) 양호 (75점)

API 토큰 비용 매우 절감 (20점)

API 지연 시간 (Latency) 매우 빠름 (15점 / 약 2~3초)
⚡ 단 1회의 LLM 호출로 전체 롤플레이 및 결과 도출 [Single API Call Payload] - "너는 가상의 방장 에이전트다. 민수, 서연, 준호 3명의 취향 카드를 분석하여 이들이 서로 대화하여 일정을 확정해가는 10줄 내외의 조율 로그를 시뮬레이션해 보여주고, 합의된 최종 결과 리스트를 JSON 형태로 반환하라." ※ 모바일 PWA 환경에서 실시간 렌더링 시 최적의 사용자 경험 제공!

3 최종 일정 검증 (Harness Linter) 및 사용자 수정 반영

에이전트들이 합의한 일정 초안, 혹은 사용자가 화면에서 직접 특정 스팟을 교체하거나 수동으로 수정할 때, 비즈니스 안전망인 TravelHarness가 즉시 작동하여 물리 조건 위반 여부를 검증하고 자가 교정 피드백 루프를 가동합니다.

🔧 사용자 수동 Tweak 시뮬레이션

사용자가 원래 예정된 식당을 삼척 갈남항에서 75km 떨어진 '강릉 대게 전문점'(메뉴 단가 95,000원)으로 수동 수정하는 시나리오입니다. 버튼을 눌러 하네스 작동을 확인해 보세요.

🔗 아바트립 백엔드 전체 시스템 파이프라인 아키텍처

실제 운영 환경에서 끊김 없고 지연 시간 없는 사용자 경험을 위해 구축되는 전체 분산 환경 파이프라인 설계도입니다. PWA ➡️ FastAPI 비동기 서버 ➡️ Redis 큐 ➡️ Celery 백그라운드 워커 ➡️ PostgreSQL + PostGIS의 결합체입니다.

📱 1. 사용자 PWA 클라이언트 Next.js App Router

3명의 친구가 공동 여행방에서 각자의 취향을 선택한 후 일정 생성 요청을 발송합니다. 결과 도출 시까지 비동기 로딩 스피너를 유지하며, 백엔드로부터 전송되는 SSE(Server-Sent Events) 이벤트를 실시간 리스닝합니다.

⬇️
🚀 2. FastAPI 게이트웨이 및 API 라우터 FastAPI (Async)

요청을 접수하면 데이터베이스 트랜잭션을 최소화하기 위해 사용자 성향 카드를 읽은 후, 즉시 백그라운드 작업으로 넘깁니다. 클라이언트에게는 202 Accepted 응답과 SSE 스트리밍 엔드포인트 커넥션을 제공하여 무한 블로킹을 차단합니다.

⬇️
⚡ 3. 메시지 브로커 & 세션 상태 저장소 Redis Pub/Sub

협상 태스크를 큐에 넣고 대기시킵니다. 또한 협상 중인 세 명의 에이전트 대화 임시 기록 및 수동 수정 진행 상태(State)를 고속 메모리에 저장하여 데이터베이스 I/O 낭비를 방지합니다.

⬇️
🤖 4. AI 에이전트 및 하네스 비동기 오케스트레이터 Celery Task Worker

실제 연산이 실행되는 엔진룸입니다. Gemini API를 통해 에이전트 협상을 모킹하고, 로컬 비즈니스 코드와 TravelHarness Linter 모듈을 활용해 동선 및 가격을 사칙연산 연산합니다. 조건 미달 시 자가 교정(Self-Correction) 프롬프트를 에이전트에게 피드백하는 루프를 격리된 워커 환경에서 최대 3회 실행합니다.

⬇️
🗺️ 5. 지리 공간 및 사용자 성향 데이터 저장소 PostgreSQL + PostGIS

공간 인덱싱 기능(SP-GIST/GIST)을 활용해 Kakao Map API를 매번 부르지 않고도 로컬 동해안 관광지 간 최단 지리적 거리(ST_Distance) 연산을 수행하며, 최종 확정된 완성 일정표의 공간 루트 정보(LINEMSTRING 기하 정보)를 데이터에 영구 보존합니다.
